# Презентация CarMatch (3–5 минут)

**CarMatch** — веб-приложение для подбора автомобиля через диалог с ИИ: пользователь описывает желания в свободной форме, система уточняет параметры, ищет подходящие машины и предлагает варианты.

---

## 1. Фронтенд

- **React 18** — интерфейс (компоненты, состояние, роутинг).
- **TypeScript** — типизация и меньше ошибок на этапе разработки.
- **Vite** — сборка и dev-сервер (быстрый старт и HMR).
- **React Router** — маршрутизация (страницы входа, чата, админки и т.д.).
- **TanStack React Query** — запросы к API, кэш и обновление данных.
- **Axios** — HTTP-клиент к бэкенду.
- **Lucide React** — иконки в едином стиле.

Фронт общается с бэкендом по REST API (авторизация JWT, эндпоинт чата и др.).

---

## 2. Бэкенд

- **FastAPI** — REST API, автодокументация (Swagger), валидация через Pydantic.
- **Uvicorn** — ASGI-сервер для запуска приложения.
- **SQLAlchemy 2** — ORM и работа с БД.
- **Alembic** — миграции схемы БД.
- **Pydantic Settings** — конфигурация из переменных окружения (.env).
- **python-jose + bcrypt** — JWT-токены и хеширование паролей.
- **httpx** — HTTP-клиент для вызовов внешних API (Яндекс, при необходимости GigaChat и др.).

Админ-API: управление пользователями, автомобилями и сессиями чата (с проверкой роли администратора).

---

## 3. База данных

- **PostgreSQL** — основная СУБД (пользователи, сессии чата, сообщения, справочник автомобилей).
- **pgvector** — расширение для хранения и поиска векторных эмбеддингов (колонка `embedding` в таблице `cars`).

Основные сущности: пользователи (`users`), сессии чата (`sessions`), сообщения (`chat_messages`), автомобили (`cars`). У машин есть текстовые поля (марка, модель, описание и т.д.) и вектор `embedding` размерности 256 для семантического поиска.

---

## 4. Поиск

Реализован **семантический (векторный) поиск** по каталогу автомобилей:

- **Эмбеддинги** — Yandex Cloud Foundation Models API:
  - для описаний машин используется модель **text-search-doc** (заполнение колонки `cars.embedding`);
  - для запроса пользователя — **text-search-query** (получение вектора запроса).
- **Хранение и поиск** — в PostgreSQL с расширением **pgvector**: косинусное расстояние (`<=>`), выбор топ-N наиболее близких по смыслу машин.
- **Гибридный подход** — векторный поиск даёт кандидатов; поверх применяются фильтры по извлечённым параметрам (бренд, тип КПП, тип кузова и т.д.) и ранжирование по степени соответствия запросу. Итоговый список передаётся в LLM для формулировки ответа пользователю.

Таким образом, поиск — это не просто «по ключевым словам», а по смыслу фразы (например, «надёжный седан для семьи»).

---

## 5. Модель LLM

Для генерации ответов в чате подбора авто используется **каскад провайдеров**:

- **Основная модель — YandexGPT** (Yandex Cloud Foundation Models):
  - в коде используется вариант **yandexgpt-lite/latest** — быстрее и экономичнее;
  - те же учётные данные, что и для эмбеддингов: `YANDEX_FOLDER_ID` и `YANDEX_API_KEY`.
- **Резервный вариант — GigaChat** (Sber): при недоступности или пустом ответе Yandex вызывается GigaChat (`GIGACHAT_CREDENTIALS`).

LLM используется для: классификации реплик (small talk / подбор авто), извлечения параметров из диалога, уточняющих вопросов, выбора и представления подходящих автомобилей из топ-кандидатов (RAG-подход: в промпт подставляются реальные данные из БД).

---

## Итог

**CarMatch** — полноценный стек: React + TypeScript + Vite на фронте, FastAPI на бэкенде, PostgreSQL + pgvector для данных и векторного поиска, Yandex (эмбеддинги + YandexGPT) для семантики и диалога, с резервом на GigaChat. Поиск машин идёт по смыслу запроса, а не только по ключевым словам.
